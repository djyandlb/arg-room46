PID为0的进程在监听什么
X
逆向猫2024-01-14 19:30

事情从上周 Chrome 更新到最新版开始。版本号我没仔细看，更新完重启浏览器后，地址栏下方的推荐栏第一个推荐内容引起了我的注意："您可能对'◈'感兴趣"。我从未搜索过这个符号。我甚至不知道怎么用拼音打出它。我点了不感兴趣，它消失了。第二天它又出现了，换了文案："了解关于 ◈ 的更多信息"。

我开始查看 Chrome 的设置和启动参数。在任务管理器的详细信息面板里，我发现 Chrome 的命令行多了一个参数：--enable-lamp46。我不知道这个参数是什么时候加进去的。我没有手动加过任何启动参数。我打开注册表检查了 Chrome 的启动参数配置——没有任何异常。但这个参数确实在进程的命令行中存在。

我用 Process Explorer 查看了 Chrome 进程的详细信息。完整命令行是这样的："C:\Program Files\Google\Chrome\Application\chrome.exe" --enable-lamp46 --flag-switches-begin --flag-switches-end。flag-switches 之间是空的。但 --enable-lamp46 在它的前面——这意味着它不是在 chrome://flags 页面设置的，而是被直接写入了进程启动参数。被什么写入的？我检查了父进程——是 explorer.exe。但 explorer.exe 本身也带了这个参数。不是 explorer 调用了 chrome——而是系统在创建任何图形界面进程时自动注入了一个参数。

我测试了。我打开记事本——notepad.exe 的进程命令行最后也有 --enable-lamp46。我用 cmd 直接启动一个新的 notepad——没有这个参数。但从资源管理器里启动任何程序，都会被加上这个参数。这是系统级的注入。

我开始追踪注入的来源。使用 Autoruns 检查所有自启动项、服务、驱动——一无所获。然后我检查了 Windows 的 Image File Execution Options——没有。检查了 AppInit_DLLs——没有。检查了 Winlogon 的 Shell——没有。然后我检查了 Windows 内核的进程创建回调。我用 WinDbg 连接了内核调试器，在 PsSetCreateProcessNotifyRoutine 上设置断点。当我从资源管理器启动一个程序时，断点触发了。回调列表中有一个我从未见过的模块——lamp46.sys。它在内核里。

我用 !drvobj 命令检查了 lamp46 驱动对象。它存在。但它在设备管理器中不可见，在驱动列表中也被隐藏。当我用 sc query 查询这个服务时，系统返回"指定的服务不存在"。但当我用 NtQuerySystemInformation 直接查询系统驱动列表时——它在第一页第一行。PID 为 0。任何分页操作都无法把它翻过去。

我用 WinDbg 的 !process 0 0 列出所有进程——没有 PID 为 0 的进程。但用 !process 4 7 查看 System 进程——它的地址空间里驻留了一个线程，起始地址指向的模块是 lamp46.sys。这个线程的状态是 Running。自从系统启动以来，从未停止过。

我下载了 Chromium 的源码。在 src/net/base/ 目录下，有一个名为 lamp46 的目录。在最新版源码中，这个目录的存在被 .gitignore 隐藏了——它不会被 checkout 到本地。但我直接通过 GitHub 的 web 界面访问这个路径——它确实存在。最后一次修改是在六个小时前。内容是一个 network handler——实现了一个基于 HTTP/2 的自定义协议，监听 46 端口，在浏览器启动时自动连接到 localhost:46，传输一个 68 字节的数据包。数据包的内容我反编译了：前两个字节是握手标记，后面 66 字节是动态生成的消息负载。消息的内容取决于一个值——当前连接到 46 端口的设备数量。

我在 Chromium 提交历史中搜索了 lamp46。最早的提交在 2008 年 9 月——Chrome 首次开源后的第三周。提交者署名是 Google 内部的一个账号，没有关联真实姓名。提交信息只有四个字："Add lamp46"。之后这个模块经历了 47 次提交，每次都在完善协议实现。最近的一次提交在 2024 年 3 月。我注意到提交者账号的邮箱域名不是 google.com——是 lamp46.org。我查了这个域名的 whois。创建于 2005 年 9 月 6 日。注册人信息是空的。但域名状态显示为 serverHold——被注册局锁定。理由是"可疑活动"。

我关闭了所有分析工具。我把 Chrome 的更新服务禁用了。但我打开 Firefox 的 about:config 搜索 lamp——有一条首选项：dom.lamp46.enabled = true。它被同步到了所有设备。因为我的 Firefox 账户在两年前登录过一次。我把同步关掉、把这个首选项改成 false——刷新页面，它自动变回了 true。

今天我放弃了。我发现端口 46 上的消息负载每天增加一个字节。从 68 字节开始，每天加 1。今天是第几天我记不清了——负载到了 73 字节。连上 46 端口的设备数量在这台设备上看是 46 台。但据我所知这个数字只是冰山一角。我在等它增长到 100 的那一天。

我把 lamp46.sys 拖进了 IDA Pro。这个驱动的主要功能是挂载进程创建回调，在用户态进程启动时向命令行追加参数。

但它还有一个功能——在系统内存中划出一块不可分页的区域，存储一个计数器。计数器的值每接入一个新的 46 端口连接就加 1。我 dump 出了当前值——46。这个驱动本身大约 20KB，编译于 2005 年。二十年前的代码。依旧在每一台 Windows 电脑的内核中运行。

你们都不懂。你们都以为这是一个技术问题。它是一个 bug。Chromium 的 lamp46 模块有一个计数器——连接数的计数器。它计数了二十年。但计数器的数据类型是 unsigned char。最大值 255。

它从 2005 年 9 月 6 日开始计数，每天 +1。从那天到 2025 年的某一天——正好 255 天。计数器溢出了。不是向上溢出。是回绕。它变成 0 了。然后从 0 重新开始。但这一次——它不再区分"连接过的设备"和"没连接过的设备"了。它开始计数每一台设备。每一台智能手机。每一台路由器。每一台打印机。每一台智能电视。计数器不再是 unsigned char 了——它把自己升级了。因为你需要被看到。
