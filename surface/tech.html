技术宅阿明 #1 · 2008-10-16 22:08

好吧我先说说我的配置：Windows XP SP3，Intel Core 2 Duo E6300，2GB DDR2 内存。不算好但也不算太差吧？之前用 Chrome 一直挺顺畅的，就是最近更新到 0.3.154.9 版本之后开始卡得不行。

具体症状：

1. 开了五六个标签页之后，Chrome 的内存占用直接飙到 1GB 以上。我刚才试了一下，只开了 Gmail、Google Reader、两个百度知道页面和这个论坛——就六个页面——任务管理器里 chrome.exe 进程的内存占用加起来有 1.2GB。我总共才 2GB 内存啊，一个浏览器吃掉一半多，剩下那点内存连开个 Word 都卡。

📷 [img/chrome-memory-taskmanager.jpg] — Windows XP 任务管理器截图，Chrome 相关进程共占 1,248MB 内存，页面文件使用量 1.8GB，CPU 使用率 65%，列表中有多个 chrome.exe 条目，经典的 XP 蓝灰界面风格

2. 滚动卡顿。不只是那些 JS 很多的页面卡，就连纯粹的文字页面，滚轮往下翻的时候也能感觉到明显的掉帧，一卡一卡的。我一开始还以为是鼠标的问题，换了个鼠标也一样。

3. 切换标签页延迟。以前点一下标签页就秒切过去了，现在点完之后有时候要等一两秒才会跳转，中间浏览器窗口会白一下。尤其是从视频网站切回来的时候，整个窗口直接白屏，过三四秒才恢复。

4. 打开新标签页白屏。按 Ctrl+T 打开新标签页，要白屏大概四五秒才能显示出来。那个"最近关闭的标签页"列表有时候是空白的，要等很久才渲染出来。

我已经试过的方法：

- 清除浏览器缓存和历史记录——无效。
- 进 chrome://settings 重置所有设置——无效。
- 卸载重装 Chrome——无效，装完还是一样卡。
- 把所有扩展都禁用了——我本来就只装了 AdBlock 和 谷歌笔记本，全禁了没变化。
- 用 CCleaner 清理了系统垃圾和注册表——无效。
- 扫描了病毒和恶意软件——没扫出东西。

现在我在怀疑是不是 Chrome 新版的内存管理机制有问题。我记得之前的版本没这么吃内存。0.2 系列的时候我开七八个页面也就占用三四百兆，现在翻了三倍。查了一些国外论坛，有人说新版本的 V8 引擎更新导致内存泄漏，也有人说是因为 Chrome 改用了新的沙箱机制所以开销变大了。但我不确定。

有碰到同样情况的朋友吗？你们都怎么解决的？还是说只能换浏览器了？

—— 技术宅阿明 · 代码改变世界 ——

蓝色大海 #2 · 2008-10-17 08:42

同遇到这个问题。我是 9 月份更新完 Chrome 之后开始卡的，情况和楼主几乎一模一样。开五六个页面就卡得不行，内存占用一眼望去全是 chrome.exe。切换标签页白屏、滚动卡顿——全中。

我折腾了两周，最后放弃了，换了 Firefox 3.0.3。说实话换之前我也不太情愿，毕竟 Chrome 刚出来的时候吹得那么好，极简界面加 V8 引擎，当时觉得这就是浏览器的未来形态。但是用起来卡就是卡，再好的理念也顶不住实际体验。

换了 Firefox 之后，同样开 Gmail + Google Reader + 三四个普通网页，内存占用稳定在 250MB 左右，滚动流畅，切换标签页秒切。Firefox 3 的酸味引擎确实比 2 代强太多了，而且 Firefox 的扩展生态比 Chrome 成熟得多——Chrome 扩展也就那几十个，Firefox 那边什么 Firebug、Web Developer Toolbar、NoScript，叫得上名字的开发工具全都有。

当然 Firefox 也不是完美的——启动速度比 Chrome 慢一点，界面也不如 Chrome 清爽。但稳定性和内存管理真的是目前最好。我建议楼主如果实在找不到解决办法，可以先投奔 Firefox 观望一下，等 Chrome 后面几个版本修了内存问题再回来。反正书签可以同步，不冲突。

对了，还有个小技巧：如果你只是要暂时给 Chrome 降内存，可以去 chrome://plugins 禁用一些不常用的插件，比如 Java 和 Adobe Reader 的插件，这些插件会被自动加载到每一个标签页，很占资源。我禁用了之后大概能减少 200MB 左右的占用，但仍然是杯水车薪。

—— 蓝色大海 · 自由撰稿人 ——

程序员小张 #3 · 2008-10-17 10:15

我大概分析了一下，Chrome 新版变卡的原因可能有好几个方面叠加。

第一，V8 引擎的更新。 0.3.x 版本的 V8 引入了新的编译器优化，按理说 JS 执行速度应该更快，但是优化有时候是以内存为代价的。V8 为每个页面分配的虚拟机实例现在会在启动时预先申请更多的内存池，而且有些优化路径的实现有内存泄漏的 Bug。我在 Chromium 的 Issue Tracker 上看到有人报告过这方面的问题，比如在大量 DOM 操作之后 V8 不会及时释放不再使用的对象。

第二，沙箱机制的资源开销。 Chrome 的沙箱设计是每个标签页一个独立进程，好处是一个页面崩溃不会影响其他页面，坏处就是每个进程都有自己的内存开销。0.3 版本之后沙箱加入了更多的安全隔离策略，新增了一些对 Windows 作业对象和访问令牌的操作，这些在内存和句柄层面都有额外消耗。你在任务管理器里看到的每个 chrome.exe 进程，有一套完整的 Windows 内核对象副本——句柄表、线程堆栈、进程环境块——这些叠加起来很可观。

第三，扩展的兼容性问题。 我注意到楼主装了 AdBlock Plus for Chrome。这个扩展在 Chrome 0.3.x 版本上有已知的性能问题——它的过滤规则引擎在新的 V8 版本上触发了某些低效率的路径，尤其是在页面加载的初始阶段，AdBlock 会遍历所有 HTTP 请求的 URL 去匹配规则列表，每个请求会产生大量的正则匹配操作。Chrome 的每个标签页有自己的进程，所以这个开销会在每个标签页上单独发生一次。建议楼主先把 AdBlock 禁用了观察一下，如果还是卡再用其他扩展排查。最好是把所有扩展全禁了，开一个干净的 Chrome 跑一整天看看。

第四，硬盘 I/O 的影响。 这个是很多人忽略的。Chrome 0.3.x 的磁盘缓存机制做了改动，现在会在后台更频繁地读写缓存数据库。如果用的是老式机械硬盘，IO 压力大的时候（比如你同时打开多个页面，每个页面都在读写自己的缓存文件），确实会感觉到卡顿。这个在 SSD 上不明显，但机械盘用户可能体会比较深。

楼主可以试试在 chrome://flags 里把 "GPU 加速渲染" 关掉，这个功能在 XP 上兼容性不太好，有可能会造成滚动卡顿。另外也可以试试在快捷方式的目标后面加上 --disable-javascript-harmony-shipping 参数来禁用一些实验性的 JS 特性，有时候能减少额外开销。

不过说到底，这些只是延寿手段，治标不治本。Chrome 团队现在还在快速迭代阶段，内存管理确实不是他们当前版本的重点。如果不急的话可以等等 0.4 或者 1.0 正式版——到时候大版本重构应该会解决这些问题。如果急的话就先换 Firefox 吧，不丢人。

—— 程序员小张 · 在改 Bug 的路上 ——
